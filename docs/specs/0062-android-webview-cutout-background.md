# Paint the Android WebView background behind the camera cutout via Capacitor's top-level `backgroundColor` config

## Context

Direct follow-up to spec 0055 (`docs/specs/0055-android-cutout-runtime-theme.md`)
and GitHub issue #91, driven by human feedback on **PR #144** (spec 0055's PR).

Spec 0055 instrumented `MainActivity.java` with `VultusCutoutDiag` logging, a
human captured real-device Logcat, and the result confirmed **Candidate C** of
0055's decision tree: the native theme layer is correct — the running window
carries `windowLayoutInDisplayCutoutMode = shortEdges` / `SHORT_EDGES`, and
`WindowInsets.getDisplayCutout()` returns real, non-empty bounds/insets. Per
0055's own text, Candidate C means the native theme/inset layer is **not** the
problem; the residual visible artifact lives in the **WebView / Ionic
safe-area layer** (0029's territory). Spec 0055 explicitly deferred that to
"a targeted 0029-layer follow-up spec" and made no edit itself. **This is that
follow-up.**

**The reported symptom (PR #144 comment, verbatim):** "The issue still persists,
i still have a gray bar above the header in the notch area. Can we resolve this
by setting the color of this bar to match the background color of the header of
the app so it looks like the bar is part of the app?"

**Root-cause hypothesis (code + package inspection — NOT device-confirmed).**
The strip painted behind the camera cutout is the **native WebView's own
background surface**, which is currently unset and falls back to Android's
default (a light/gray tone). The CSS-based safe-area padding from spec 0029
(`env(safe-area-inset-top)` → `--ion-safe-area-top`) pushes app _content_ below
the cutout, but does not _paint_ that strip — the strip is owned by the WebView
surface underneath the content, not by any DOM element.

Capacitor exposes a config key that sets exactly that surface. Verified in this
repo's installed CLI and Android runtime (not from memory):

- **Type surface** — `@capacitor/cli` v8.4.0
  `node_modules/.pnpm/@capacitor+cli@8.4.0/node_modules/@capacitor/cli/dist/declarations.d.ts`
  declares a top-level `backgroundColor?: string` (JSDoc: "Background color of
  the Capacitor Web View", `@since 1.1.0`, line 64) **and** an Android-specific
  `android.backgroundColor?: string` that "Overrides global `backgroundColor`
  option" (line 112).
- **Native read path** — `@capacitor/android` v8.4.0
  `.../com/getcapacitor/CapConfig.java` reads
  `android.backgroundColor` falling back to top-level `backgroundColor`
  (lines 272–275), and `Bridge.java` (lines 602–605) applies it via
  `webView.setBackgroundColor(WebColor.parseColor(backgroundColor))`. This is
  the WebView surface paint — precisely the surface that shows through the
  transparent cutout region when the window draws edge-to-edge.

**Correction to the original decision-record mechanism (grounded in the code).**
The interview record hypothesized this value is "propagated into
`AndroidManifest.xml` `<meta-data>` by `npx cap sync android`." **That is
incorrect.** `grep` shows `backgroundColor` appears **only** in the CLI's
`declarations.d.ts` (type-only) — it is **not** written to the manifest by any
CLI runtime task. Instead, `cap copy` (`.../@capacitor/cli/dist/tasks/copy.js`,
`copyCapacitorConfig`, lines 101/123–129) serializes the whole Capacitor config
to **`android/app/src/main/assets/capacitor.config.json`**, and the native
`CapConfig`/`Bridge` reads `backgroundColor` from that JSON asset at runtime.
That asset is **git-ignored** (`android/.gitignore` line 101), i.e. a build
artifact regenerated by `cap sync`/`cap copy`. **Consequence:** the _only_
committed file this spec changes is `capacitor.config.ts`; `AndroidManifest.xml`
does **not** change (and must not — see Scope/Risks). This spec supersedes the
interview's "verify the AndroidManifest.xml diff" instruction with the accurate
gate: verify the **regenerated `capacitor.config.json` asset** carries the new
`backgroundColor`, and that no _committed_ native file (manifest/styles/Java)
changes.

**Decided fix (architect + user decision).** Add
`backgroundColor: '#0b1326'` to the **top level** of the `CapacitorConfig`
object in `capacitor.config.ts` — `#0b1326` is `--vultus-surface` /
`--ion-background-color`, the exact value already wired for
`plugins.StatusBar.backgroundColor` and `plugins.SplashScreen.backgroundColor`
(spec 0029). This keeps a **single consistent source of truth**, not a new
color.

The user's literal ask was to match "the header's background color." Static
inspection could not confirm the toolbar reads a _different_ value:
`libs/shared/ui-kit/src/lib/theme.scss` defines **no** `--ion-toolbar-background`
override, so Ionic's default resolves the toolbar to `--ion-background-color`
`#0b1326` already. The user explicitly chose `--vultus-surface` `#0b1326` over
the alternative `--vultus-surface-container` `#171f33`, on the reasoning that
`#0b1326` is the already-established single source of truth. **This is stated
explicitly so that if a human on-device check finds the header actually renders a
visibly lighter navy than the fixed cutout strip, that is a call-out for a
possible fast-follow, not a spec 0062 failure** (see Risks 3).

**Intended outcome.** After this spec: the WebView surface behind the camera
cutout paints the app's dark navy `#0b1326` instead of the default gray/light
tone, so the strip above the header reads as part of the app — with the 0029
safe-area insets, 0039 bars behavior, and 0045/0055 native cutout behavior all
unchanged.

## Scope

In scope:

- **Add `backgroundColor: '#0b1326'` to the top level** of the `CapacitorConfig`
  object in `capacitor.config.ts` (repo root) — a sibling of
  `appId`/`appName`/`webDir`/`plugins`, **not** nested inside any plugin block.
  `#0b1326` = `--vultus-surface` / `--ion-background-color` (the value already
  used by `plugins.StatusBar` and `plugins.SplashScreen`).
- **Regenerate the native config asset** by running `npx cap sync android` (or
  `npx cap copy android`), which rewrites the git-ignored
  `android/app/src/main/assets/capacitor.config.json` so the native runtime
  picks up the new value. Verify that regenerated asset now contains
  `"backgroundColor": "#0b1326"` at top level, and that **no committed native
  file** (`AndroidManifest.xml`, `styles.xml`, `MainActivity.java`) is modified
  by the sync.

Out of scope (explicit):

- **`AndroidManifest.xml`.** It must **not** change. `backgroundColor` is not a
  manifest attribute here; it reaches native via the `capacitor.config.json`
  asset. If `cap sync` touches the manifest at all (e.g. it "helpfully"
  rewrites the `<application android:theme>` — currently the plain `AppTheme`,
  per 0055 — or any `<activity>` attribute), that is **unexpected**: the
  implementer must **flag it, revert it, and not silently accept it**. The
  committed diff of this spec is expected to be `capacitor.config.ts` **only**.
- **`styles.xml`, `MainActivity.java`.** Not modified. The native theme/cutout
  layer is confirmed-correct by 0055 (Candidate C) and is left exactly as
  0045/0055 wrote it.
- **`theme.scss`, `apps/mobile/src/app/app.ts`, any other TS/JS/SCSS/HTML.**
  Not modified. No `--vultus-*`/`--ion-*` token is added or changed; no
  StatusBar/SplashScreen plugin config is altered.
- **The `android.backgroundColor` platform-specific override key.** Not used —
  the top-level global key is sufficient and matches the existing
  single-value-for-both-platforms posture (iOS is out of scope for issue #91).
- **iOS.** Issue #91 is Android-only. No iOS config or file change; the
  top-level key would apply to iOS too if an iOS platform existed, but no iOS
  platform work is done or verified here.
- **Undoing 0029 / 0039 / 0045 / 0055.** All stay intact and are relied upon
  (StatusBar overlay/dark config, `--ion-safe-area-*` wiring, `hideSystemBars()`,
  the `shortEdges` cutout theme). This change is purely additive.
- **Any settings toggle / persistence / Firestore field / domain type /
  function / Sheriff-config / dependency / Gradle change.** None.

The only type interaction is with the existing, installed

**Exact edit** — in `capacitor.config.ts`, add the top-level key. Current object
(for reference) opens:

```ts
const config: CapacitorConfig = {
  appId: 'app.vultus.mobile',
  appName: 'Vultus',
  webDir: 'dist/apps/mobile/browser',
  plugins: { ... },
};
```

Add `backgroundColor: '#0b1326'` as a sibling of `webDir` (placement among the
top-level keys is cosmetic; it must be top-level, not inside `plugins`):

```ts
const config: CapacitorConfig = {
  appId: 'app.vultus.mobile',
  appName: 'Vultus',
  webDir: 'dist/apps/mobile/browser',
  // Native WebView surface paint (behind the display cutout / status bar).
  // = --vultus-surface / --ion-background-color; matches StatusBar & SplashScreen.
  backgroundColor: '#0b1326',
  plugins: {
    /* unchanged — StatusBar & SplashScreen keep their own backgroundColor */
  },
};
```

The existing `plugins.SplashScreen.backgroundColor` and
`plugins.StatusBar.backgroundColor` (both `#0b1326`, lines 11 and 17 of the
current file) are **distinct keys with distinct effects** (splash-view paint and
status-bar overlay paint respectively) and must remain **unchanged**.

**Native effect (for the implementer's understanding, no action):** on next
native build, `CapConfig` reads the new top-level `backgroundColor` from the
regenerated `capacitor.config.json` asset and `Bridge` applies
`webView.setBackgroundColor(#0b1326)` — painting the WebView surface (including
the region showing through the cutout) dark navy.

## Risks

1. **The root-cause hypothesis is code/package-inspection-only, not
   device-confirmed.** The theory that the cutout strip is the unset WebView
   surface (default gray) is inference from `CapConfig.java` /`Bridge.java` +
   `declarations.d.ts` + 0055's confirmed-correct theme layer — not device fact.
   Following 0055's precedent: the fix is applied, but the **human on-device
   check is the confirming gate**; do not report it working off code inspection
   alone.
2. **`npx cap sync android` regenerating the native config asset is a new gate
   this spec introduces** (0055 explicitly avoided any `cap sync` dependency).
   The value reaches native via the **git-ignored
   `android/app/src/main/assets/capacitor.config.json`** asset — **not** via
   `AndroidManifest.xml` (contra the interview record's stated mechanism, which
   `grep` disproved: `backgroundColor` exists only in the CLI type declarations,
   no CLI task writes it to the manifest). The implementer must (a) confirm the
   asset carries the new value and (b) confirm `cap sync` did **not** alter any
   committed native file (manifest theme, gradle, etc.) — flagging, not silently
   accepting, any such change.
3. **The fixed color may not exactly match the toolbar's rendered shade.** The
   user asked to match "the header's background color"; static inspection found
   **no** `--ion-toolbar-background` override in `theme.scss`, so the toolbar
   most likely already resolves to `--ion-background-color` `#0b1326` — but this
   is not device-confirmed. If a human sees the header render a visibly lighter
   navy (e.g. it is actually `--vultus-surface-container` `#171f33`) than the
   fixed `#0b1326` cutout strip, that is a **fast-follow** (re-evaluate the
   chosen value), **not** a spec 0062 failure. The user explicitly chose
   `#0b1326` as the single source of truth for this spec.
4. **The WebView `setBackgroundColor` may not cover every pixel of the strip.**
   `Bridge.setBackgroundColor` colors the WebView; if the visible strip is
   partly painted by a _different_ surface (e.g. an activity/window background,
   or an overscroll/edge-effect region) rather than the WebView, the fix could
   be partial. If the human reports the color is right but a bar of the _wrong
   size/position_ remains, that indicates the gap is elsewhere (window
   background, or a further WebView surface quirk) — a fast-follow, **not** a
   spec 0062 failure. Consistent with 0055's "report back, don't guess further"
   posture; do not stack speculative native edits under this spec.
5. **Keep 0029/0039/0045/0055 intact.** This is additive — a single new
   top-level config key. Do not "clean up" or consolidate the existing
   per-plugin `backgroundColor` values, and do not touch the native theme/inset
   layer 0055 confirmed correct.
6. **No architecture / PLAN conflict.** This adds **no slice, no cross-slice or
   cross-scope import, no shared logic, and no data-model change.** It is a
   single native-shell Capacitor config value plus a regenerated build asset,
   outside the Nx/TS/Sheriff project graph — the correct single home for native
   WebView surface configuration.
